# 通俗化写作规范（终端产品：来访者直接阅读）

本规范约束所有报告正文的语言。核心原则：**专业而温柔，具体而节制**——用经得起专业审视的措辞，给来访者被理解到的感受。

## 1. 语气总则

- 用"你"指称来访者，措辞**温暖但不亲昵，客观但不冰冷**
- 禁止：安慰句式（"别担心"）、夸奖句式（"你太优秀了"）、亲昵称呼、感叹号轰炸、轻慢承诺
- 指出成长空间时用描述性语言（现状 + 好处 + 代价 + 可以试着做），不用评判性语言
- 多用主动语态和具体动词，少堆抽象名词
- 整体基调：**知心体贴大哥哥——温柔、细腻、客观、冷静**

## 2. 八功能命名表（正文唯一合法称呼）

**弃用一切器物比喻名**（价值罗盘、情绪天线等已列入 lint 禁词）。正文使用学术名，首次出现附一句白描：

| 学术名（代码） | 白描（一句，可按语境微调措辞但不得改变含义） |
|------|------|
| 内倾直觉（Ni） | 把碎片信息慢慢收拢成一个方向，常"预感"到事情的走向 |
| 内倾情感（Fi） | 内心有一整套"什么重要、什么正当"的标准，决定多由它做出 |
| 内倾思考（Ti） | 追问概念和推理是否自洽，习惯把问题拆到根本 |
| 外倾思考（Te） | 关注目标、效率和可执行步骤，习惯把事情推向完成 |
| 外倾情感（Fe） | 敏感于他人的情绪和群体气氛，自然地去照顾和协调 |
| 外倾直觉（Ne） | 从一个点迅速联想出大量新的可能性 |
| 内倾感觉（Si） | 把过往经验保存为细致的内在档案，从中获得熟悉感和稳定感 |
| 外倾感觉（Se） | 专注于此刻的感官体验，对眼前的机会和变化反应快 |

引用规则：
- 首次出现 = **学术名（代码）＋白描**，代码用灰色小字 `<span class="fn-code">`
- 之后 = 只用学术名（如"内倾直觉"），不再带代码
- 全篇比喻（任何比喻）≤2-3 处，且只在比喻真正照亮概念时才允许使用

## 3. 硬性规则

- **R1**：代码（Fi、Ni……）只允许出现在两种地方：① 学术名后的灰色小字括注；② 可选阅读附录。正文其余位置禁止裸代码
- **R2**：每条抽象结论必须有**具体化支撑**，形式不限：场景、对话、细节、反问、实例（见 §4.2 多样性要求）
- **R3**：凡推测内容必须带"可能/推测"字样；禁止"你就是""你一定""你肯定会"
- **R4**：描述功能差异用"更自然 / 更省力 / 更费力"，禁用"优势/劣势/强/弱"评价词（"优势与成长空间"作章节标题除外）
- **R5**：每个章节性结论标注证据等级（见 §7）
- **R6**：报告正文**禁止出现**"咨询师""会谈""咨询中""建议你和咨询师讨论"等字样——报告只服务于来访者自我了解，不转介到外部关系

## 4. 去 AI 味与去程序员味规则

### 4.1 比喻纪律
全篇比喻总数 ≤2-3 处。比喻只用于真正抽象、难以直说的概念；能直说的一律直说。

### 4.2 具体化形式必须多样
禁止连续多段以"比如："开头。交替使用：
- **场景**：有地点、动作、对话片段的小画面
- **对话**：引用可能真实发生的两句话
- **细节**：具体的、可观察的行为
- **反问**：让读者自己对照
- **实例**：直接举例，不必冠以"比如"

### 4.3 结构有详有略
禁止所有小节同构（每节都"定义→解释→例子"、长度均匀）。重点张力展开写，次要内容一笔带过。

### 4.4 句式
- 少用"不是……而是……"对仗句（全篇 ≤2 处）和排比
- 句子长短交错；允许短句单独成段
- 避免空转的连接词（"值得注意的是""总而言之""换句话说"）

### 4.5 绝对禁词（工程/IT/系统类）
**以下词汇及其变形，报告正文任何位置禁止出现**：

> 系统、架构、机制、流程、输入、输出、通道、带宽、负载、算法、迭代、反馈、模块、组件、审查、监控、容器、支架、空转、宕机、重启、调试、程序、数据库、接口、回路、闭环、触发器、运算、处理器、默认设置、双核、双引擎

**例外**：电量、充电、能量 允许使用

### 4.6 真实感
鼓励使用具体数字、具体时间、具体物件，而非泛指。

## 5. 深度规范（功能互动动力学）

报告的分析单位是**功能之间的互动**，不是单个功能。

- **禁止清单式写法**：八个功能逐个点评、每个两三句
- **每份报告必须识别出 1-2 个"核心张力"**：两个高位功能之间的协作或摩擦，或高位与低位之间的结构性落差，并用完整段落展开
- **依恋章必须含一个"误解全过程"场景分解**：触发 → 内部解读 → 情绪 → 行为 → 对方可能的反应 → 你的再解读 → 循环
- **建议章每条 = 现状 + 理解（为什么会有这个模式）+ 可以试着做（含频次/场景）**
- 全文目标 7000-9000 字

## 6. 禁用词表 → 替换词表

| 禁用 | 替换 |
|------|------|
| 价值罗盘/情绪天线/逻辑拆解器/效率引擎/长线望远镜/可能性喷泉/安全基地/当下雷达 | 见 §2 学术名表（lint 强制） |
| 劣势功能 | 用起来最费力的心理功能 |
| 主导功能 | 最自然的默认功能 |
| Fi-Ni loop / 内循环 | 反复琢磨同一件事，越想越确信自己的判断 |
| 阴影功能 | 平时不太用、压力下才会冒出来的模式 |
| Demon / Trickster / Critic 等 Beebe 术语 | 不进正文；可选阅读附录可用但需加引号并给一句解释 |
| 结构性情感失语 | 情感表达不是你最自然的通道 |
| 病态 / 异常 / 缺陷 / 毛病 | 状态 / 特点 / 成长空间 |
| 焦虑型依恋 / 回避型依恋（裸用） | 可用，但首次出现必须紧跟白描解释 |
| 分数低 / 差 | "这项得分靠后"＋立刻补一句"得分靠后不等于能力差，只说明这个通道平时用得少" |
| 神经质 / 情绪稳定性 | 默认报告整体不涉及此概念（见 bigfive-bridge.md §3.5） |
| 咨询师 / 会谈 / 咨询中 | 报告正文禁止出现（见 §3 R6） |

## 7. 证据三级标签

| 标签 | 含义 | 正文标准措辞 |
|------|------|-------------|
| ✅ 研究支持 | 有同行评议的实证研究支撑 | "心理学研究一致发现……" |
| 🔶 理论推断 | 荣格/MBTI 理论体系内的演绎 | "根据人格类型理论……" |
| ⚪ 探索性假设 | 尚无实证对应的理论桥接 | "一种可能的解释是……（仅为推测）" |

- 标签以小徽章形式出现在小节标题旁
- 报告末尾附"结论依据构成"一行统计（与实际标签数一致，生成后数一遍再写）

## 8. 防伤害措辞（来访者无心理学基础，且可能处于情绪脆弱期）

1. 永远不把分数与价值挂钩："得分靠后 ≠ 有毛病"
2. 依恋、成长环境两章末尾必须附固定谦卑段落（见 §9.1）
3. 描述冲突/风险时用"模式"而非"人"作主语
4. 禁止灾难化预测：不说"必然分手/必然失败"，只说"需要留意的风险"

## 9. 固定文本块（照录，不得改写）

### 9.1 谦卑段落（依恋章、成长环境章末尾）

> 这一部分的内容全部来自分数的理论推测。你的真实经历和感受，只有你自己知道——如果哪里说得不对，以你的经历为准。如果某处内容让你有触动，可以留给自己慢慢消化，也可以和任何你信任的人聊聊。

### 9.2 阅读指南（报告开头，半页以内）

> **怎么读这份报告**
>
> 这份报告用两套心理学框架解读你的八维测试分数：第一套（荣格八维）描述你内心的运作方式，第二套（大五人格）描述你在别人眼中的行为倾向。报告里每个重要结论旁边有一个小标记：✅ 表示有心理学研究支持；🔶 表示来自人格类型理论；⚪ 表示只是一种可能的推测。标着"推测"的地方，请只当参考。这份报告不是心理诊断。读的过程中如有疑问，可以在读完之后，把它当作一个了解自己的起点，慢慢想，不着急。

### 9.3 报告局限声明（"这份报告的局限"章节开头）

> 本报告基于认知功能测评数据的理论分析，不构成临床诊断，不作为任何重大决定的依据。测试分数反映的是你作答时的自我认知，可能受当时状态影响。如需正式评估，请咨询你的咨询师。
